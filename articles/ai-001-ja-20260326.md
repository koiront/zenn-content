## はじめに — なぜ今Claude Codeなのか

2025年後半から2026年にかけて、AI駆動開発は「話題のバズワード」から「日常のワークフロー」へと完全に移行しました。GitHub Copilot、Cursor、そしてClaude Code——選択肢は増え続けていますが、その中でもClaude Codeはターミナルベースという独自のポジションで急速にユーザーを増やしています。

ただ、Web上にある記事の多くは「インストール方法」や「機能一覧」の紹介に留まっていて、**実務で毎日使ってみてどうだったか**という情報が意外と少ないと感じていました。

この記事では、自分が実際のプロジェクトでClaude Codeを日常的に使い始めてから分かった「本当に生産性が上がる使い方」と「正直イマイチだったこと」を、具体的なコマンド例付きで共有します。

:::message
この記事は2026年3月時点の情報に基づいています。Claude Codeは更新頻度が非常に高いので、最新情報は[公式ドキュメント](https://code.claude.com/docs)を確認してください。
:::

## 生産性が上がった3つのユースケース

### ① Plan Mode → Auto-acceptの2段階ワークフロー

これは一番おすすめしたい使い方です。

Claude Codeを起動したら、まず`Shift+Tab`を2回押してPlan Modeに入ります。ここで実装したいことを伝えると、Claude Codeが「こういう方針で進めます」という計画を立ててくれます。

```bash
# Plan Modeで起動
claude
# Shift+Tab × 2 でPlan Modeに切り替え

> ユーザーのプロフィールページに、最近の活動履歴を表示するコンポーネントを追加したい。
> APIは /api/users/:id/activities が既にある。
```

計画が出てきたら内容を確認して、OKなら`Shift+Tab`を1回押してAuto-acceptモードに切り替えます。すると、計画に沿ってClaude Codeが一気に実装を進めてくれます。

**なぜこれが効果的か：**

- いきなりコードを書かせると、方向性がズレたまま大量のコードが生成されるリスクがある
- Plan Modeで「合意形成」してからAuto-acceptに移行すると、手戻りが激減する
- コンテキストがクリアな状態で計画を立てるため、精度が高い

### ② git diffパイプでコードレビュー自動化

チームでPRレビューを回しているなら、これは試す価値があります。

```bash
# 差分をClaude Codeに渡してレビュー
git diff main..feature/new-component | claude -p "このdiffをレビューして。バグ、パフォーマンス問題、命名の改善点を指摘して。"
```

`-p`フラグを使うとワンショットモードで実行できるので、CIに組み込むことも可能です。

自分の場合は、PR作成前のセルフレビューとして使っています。人間のレビュアーに回す前に明らかなミスを潰せるので、レビュー全体の効率が上がりました。

### ③ CLAUDE.mdによるプロジェクトルールの永続化

Claude Codeはセッションをまたぐと記憶を失います。毎回「TypeScriptで書いて」「テストはVitestで」と言うのは非効率です。

そこで`CLAUDE.md`ファイルです。プロジェクトのルート、またはホームディレクトリに配置すると、Claude Codeが自動的に読み込んでくれます。

```markdown
# CLAUDE.md

## プロジェクト概要
Next.js 14 + TypeScript のWebアプリケーション

## コーディング規約
- 関数コンポーネントを使用（クラスコンポーネント不可）
- スタイリングはTailwind CSSを使用
- 状態管理はZustandを使用
- テストフレームワークはVitest + Testing Library

## コマンド
- `pnpm dev` - 開発サーバー起動
- `pnpm test` - テスト実行
- `pnpm lint` - Lint実行

## 重要な注意事項
- APIキーなどの機密情報をコードにハードコードしない
- コンポーネントは1ファイル200行以内に収める
```

`/init`コマンドで自動生成もできますが、自動生成されたものをベースに手動で調整するのがおすすめです。

## 逆に期待はずれだったこと・ハマったポイント

良いことばかり書いても信用されないので、正直にイマイチだったことも共有します。

### コンテキストの溢れ問題

大きなファイルを複数読み込んだり、長い対話を続けていると、コンテキストウィンドウが上限に達して精度がガクッと落ちます。

**対策**: こまめに`/clear`でコンテキストをリセットする。タスクは小さく分割して、1セッション1タスクを心がける。

### 日本語IMEとの相性

<!-- ✏️ 加筆ポイント: Ctrl+Spaceの問題など、実体験を詳しく書く -->

Windows環境で日本語IMEを使っていると、ショートカットキーが競合することがあります。特にVSCode連携時に`Ctrl+Space`がIMEに取られてしまう問題は結構ストレスでした。

### レート制限

Proプランだと、集中して使っていると5時間枠のレート制限に引っかかります。「ノッてきたところで止まる」のが地味にキツい。Max 5xにアップグレードしたらかなり改善されましたが、それでもヘビーに使うと制限に達します。

## 実践Tips 5選

### 1. テストファーストとの相性が最高

Claude Codeにいきなり実装を書かせるより、**先にテストを書かせる→テストが通る実装を書かせる**の順番が圧倒的にうまくいきます。

```
> formatCurrency関数を作りたい。数値を受け取って日本円表記の文字列を返す。
> まずVitestのテストファイルだけ作って。正常系、境界値、エラーケースを網羅して。
```

テストが生成されたら確認して、OKなら：

```
> テストの内容はOK。では、このテストがすべてパスする実装をsrc/lib/format.tsに書いて。
> 書いたら pnpm test を実行して結果を見せて。
```

テストが「仕様書」になるので、Claude Codeが迷いにくくなります。

### 2. SubAgentで調査と実装を分離する

調べ物をしながら実装するとコンテキストが膨らみます。SubAgentを使えば、リサーチを別セッションに切り出せます。

```
> Next.js 14のServer ActionsでファイルアップロードするベストプラクティスをWebSearchで調べて。
> 結果をまとめてから、実装方針を提案して。
```

Claude Codeが自動的にSubAgentを起動してリサーチし、結果をメインセッションに持ち帰ってくれます。

### 3. Hooksでフォーマッターを自動実行

`.claude/settings.json`にHooksを設定すると、ファイル編集後に自動でフォーマッターが走ります。

```json
{
  "hooks": {
    "postFileWrite": {
      "command": "npx prettier --write $FILE"
    }
  }
}
```

Claude Codeが生成したコードが自動的にフォーマットされるので、差分がきれいになります。

### 4. permissionの設定で確認ダイアログを減らす

毎回「この操作を許可しますか？」と聞かれるのがストレスなら、安全な操作は自動許可にしましょう。

```json
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "WebSearch",
      "WebFetch",
      "Read"
    ]
  }
}
```

### 5. MCP連携でNotionやSlackと繋ぐ

プロジェクトの仕様書がNotionにある場合、MCPで接続するとClaude Codeが直接参照できます。設計ドキュメントを読みながら実装できるのは非常に便利です。

## まとめ — Claude Codeとの付き合い方

1ヶ月使ってみて確信したのは、**Claude Codeは「魔法の杖」ではなく「優秀なペアプロ相手」**だということです。

うまく使うコツは：

- **コンテキストを制する**：CLAUDE.mdで前提を共有し、セッションはこまめにリセット
- **Plan → Executeの2段階**：いきなり書かせず、まず計画を立てさせる
- **テストファースト**：テストという「仕様書」を先に作って精度を上げる
- **小さく分割**：1セッション1タスクを心がける

AIツールは「使い方を知っている人」と「なんとなく使っている人」で生産性の差が大きく開きます。この記事が、Claude Codeを使いこなすヒントになれば嬉しいです。
