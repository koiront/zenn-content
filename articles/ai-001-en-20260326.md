## Three use cases that genuinely improved productivity

### 1. Plan Mode → Auto-accept two-phase workflow

This is my top recommendation.

Launch Claude Code, press `Shift+Tab` twice to enter Plan Mode. Describe what you want to build, and Claude Code creates an implementation plan.

```bash
# Start Claude Code
claude
# Shift+Tab × 2 to switch to Plan Mode

> I want to add an activity history component to the user profile page.
> The API endpoint /api/users/:id/activities already exists.
```

Review the plan, and if it looks good, press `Shift+Tab` once to switch to Auto-accept mode. Claude Code then executes the plan end-to-end.

**Why this works:**

- Jumping straight to code generation risks going in the wrong direction
- "Agreeing on the plan" before auto-accepting drastically reduces rework
- Planning on a clean context produces more accurate results

### 2. Automated code review via git diff pipe

If your team does PR reviews, this is worth trying.

```bash
git diff main..feature/new-component | claude -p "Review this diff. Flag bugs, performance issues, and naming improvements."
```

The `-p` flag runs Claude Code in one-shot mode, making it easy to integrate into CI pipelines.

I use this as a self-review step before opening a PR. Catching obvious issues early makes the entire review process faster.

### 3. CLAUDE.md for persistent project rules

Claude Code loses memory between sessions. Repeating "use TypeScript" and "tests are in Vitest" every time is inefficient.

Place a `CLAUDE.md` file in your project root:

```markdown
# CLAUDE.md

## Project overview
Next.js 14 + TypeScript web application

## Coding conventions
- Functional components only (no class components)
- Styling with Tailwind CSS
- State management with Zustand
- Testing with Vitest + Testing Library

## Commands
- `pnpm dev` - dev server
- `pnpm test` - run tests
- `pnpm lint` - run linter

## Important
- Never hardcode API keys or secrets
- Keep components under 200 lines per file
```

You can auto-generate one with `/init`, then tweak it manually.

## What didn't work — honest takes

### Context overflow

When you load multiple large files or have a long conversation, the context window fills up and accuracy drops noticeably.

**Mitigation**: Use `/clear` frequently. Keep tasks small — one task per session.

### Japanese IME conflicts

On Windows with Japanese IME, keyboard shortcuts can conflict. The `Ctrl+Space` shortcut being captured by IME instead of VSCode was particularly frustrating.

### Rate limits

On the Pro plan, focused usage can hit the 5-hour session rate limit. Upgrading to Max 5x helped significantly, but heavy usage can still trigger limits.

## Five practical tips

### 1. Test-first pairs perfectly with Claude Code

Instead of asking Claude Code to write implementation directly, ask for tests first, then implementation.

```
> I want a formatCurrency function that takes a number and returns a formatted USD string.
> First, write only the Vitest test file. Cover happy paths, edge cases, and error cases.
```

After confirming the tests:

```
> Tests look good. Now write the implementation in src/lib/format.ts that passes all tests.
> Then run pnpm test and show me the results.
```

Tests act as a specification, reducing ambiguity for Claude Code.

### 2. Use SubAgents to separate research from implementation

Research bloats context. SubAgents run in a separate session.

```
> Research best practices for file uploads with Next.js 14 Server Actions using WebSearch.
> Summarize the findings, then propose an implementation plan.
```

### 3. Auto-format with Hooks

Set up Hooks in `.claude/settings.json` to run formatters after file edits:

```json
{
  "hooks": {
    "postFileWrite": {
      "command": "npx prettier --write $FILE"
    }
  }
}
```

### 4. Reduce confirmation dialogs with permissions

```json
{
  "permissions": {
    "allow": ["WebSearch", "WebFetch", "Read"]
  }
}
```

### 5. MCP integration with Notion and Slack

If your specs live in Notion, connect it via MCP so Claude Code can reference them directly during implementation.

## Takeaway — How to work with Claude Code

After a month of daily use, I'm convinced that **Claude Code is not a magic wand — it's an excellent pair-programming partner**.

Keys to making it work:

- **Manage context**: Share baseline knowledge via CLAUDE.md, reset sessions frequently
- **Plan before executing**: Don't jump to code — let it plan first
- **Test-first**: Create a test-based "spec" to improve accuracy
- **Keep tasks small**: One task per session

The gap between developers who understand how to use AI tools and those who use them casually is widening fast. I hope this article helps you get more out of Claude Code.
